Merkmale

  • Abstraktion der physischen Infrastruktur
  • Reduzierter Verwaltungsaufwand
  • Hohe Verfügbarkeit und Ausfallsicherheit
  • Nutzungsbasierte Abrechnung
  • Automatische Skalierung (Elastische Skalierung)
  • Skalierung auf Null
{% include padding-id.html id=2 %}

Herausforderungen

  • Kaltstart und potenziell höhere Latenz
  • Sicherheit - weniger direkte Kontrolle
  • Anbieterbindung (Vendor Lock-in)
  • Begrenzungen hinsichtlich der maximalen Ausführungszeiten
  • Begrenzter Zugriff auf zentrale Infrastrukturaspekte
  • Debugging und Fehlersuche
  • Kostenunvorhersehbarkeit
  • Auswahlkomplexität
{% include padding-id.html id=3 %}

Automatische elastische Skalierung

  • Anpassung von Ressourcen (VMs, Container, Rechen-/Speicherressourcen)
  • Automatisch in Echtzeit an aktuelle Anforderungen
  • Bewältigung großer Transaktionsvolumen und Lastspitzen ohne manuelle Eingriffe
  • Kontinuierliche Überwachung zentraler Leistungsmetriken (CPU, Speicher, Netzwerk, Anfragen) in Echtzeit
  • Entscheidung über Ressourcenzuweisung basierend auf vordefinierten oder maschinell erlernten Regeln/Schwellenwerten
  • Inkrementelle Skalierung
  • Automatischer Skalierungsprozess
{% include padding-id.html id=4 %}

Skalierung auf null (Scale-to-Zero)

  • Ressourcen automatisch vollständig auf null herunterzufahren, wenn keine Anfragen oder Aktivitäten an die Datenbank gesendet werden
  • Sobald eine neue Anfrage eingeht, wird die Datenbank automatisch wieder hochgefahren und die erforderlichen Ressourcen bereitgestellt
  • Basiert häufig auf einer verteilten Architektur über mehrere Knoten
  • Rechenkapazität (CPU, RAM) und Speicherkapazität können unabhängig skaliert werden
  • Bei minimaler Aktivität werden aktive Knoten heruntergefahren und temporäre Daten verschoben oder gelöscht. Netzwerkressourcen (Verbindungen, Load Balancers) werden deaktiviert
  • Bei neuer Anfrage: System erkennt Aktivität, startet "Hochskalierung" von Null
  • Erforderliche Knoten werden neu gestartet, Rechenressourcen zugewiesen
  • Prozesse sind auf schnelle und reibungslose Wiederaufnahme ausgelegt, um Verzögerungen zu minimieren
  • Architektur ist optimiert, damit keine Daten während der Skalierung verloren gehen
{% include padding-id.html id=5 %}

Preismodell - Pay-As-You-Go

  • Abrechnungsmodell, bei dem Kunden nur für die tatsächlich genutzten Ressourcen, Dienste oder Produkte bezahlen
  • Kosten sind direkt an das Volumen oder die Menge der verbrauchten Ressourcen gekoppelt (z. B. Anzahl der Anfragen, Speicherkapazität)
  • Keine hohen Anfangsinvestitionen oder langfristigen Verträge notwendig
  • Ermöglicht flexible Nutzung ohne langfristige finanzielle Verpflichtungen
{% include padding-id.html id=6 %}

Preismodell - Provisionierung

  • Bezeichnet die Zuweisung und Reservierung einer festen Menge an IT-Ressourcen im Voraus
  • Für Nutzer, die eine konstante Arbeitslast und Leistungsniveaus erwarten, bieten einige serverlose Datenbanken diese Option
  • Vorteil: Verfügbarkeit und Leistung der Ressourcen sind sichergestellt, unabhängig von der tatsächlichen Nutzung
  • Kosten: Im Voraus festgelegt und bleiben konstant, unabhängig von der Nutzung
  • Kann für Anwendungen mit konsistenter, vorhersehbarer Leistung kostengünstiger sein
  • Viele Anbieter bieten dennoch Flexibilität zur Anpassung/Erweiterung der provisonierten Ressourcen bei Bedarf
{% include padding-id.html id=7 %}

Sicherheit

  • Sicherheitsaspekte spielen eine entscheidende Rolle, da die Infrastruktur-Verantwortung beim Cloud-Anbieter liegt
  • Cloud-Anbieter implementieren umfassende Sicherheitsmaßnahmen zur Gewährleistung der Datenintegrität
  • Zentrale Sicherheitsaspekte:
    • Datenverschlüsselung: Schutz von Daten während Übertragung und im Ruhezustand
    • Zugriffskontrolle: Rollenbasierte Zugriffskontrolle und Identitäts-/Zugriffsmanagement (IAM)
    • Netzwerksicherheit: Virtual Private Clouds (VPCs) und Netzwerksicherheitsgruppen zur Isolierung
    • Protokollierung und Überwachung: Nachverfolgung von Zugriffen/Änderungen für Compliance und Sicherheitsvorfälle
    • Datenisolierung: Strikte Trennung der Daten verschiedener Kunden in Multi-Tenant-Umgebungen
{% include padding-id.html id=8 %}

Verfügbarkeit und Zuverlässigkeit

  • Darauf ausgelegt, eine kontinuierliche Nutzung von Anwendungen sicherzustellen, auch unter unerwarteten Bedingungen
  • Automatische Replikation der Daten über mehrere Rechenzentren (Availability Zones)
  • Fehlererkennungs- und -behebungsmechanismen überwachen kontinuierlich, erkennen Fehler automatisch und leiten Verkehr bei Bedarf auf funktionierende Instanzen um, bevor ein Ausfall spürbar wird
  • Integration von Backup- und Wiederherstellungsfunktionen ermöglicht schnelle Datenwiederherstellung
  • Redundanz und Lastverteilung tragen dazu bei, dass die Datenbank funktionsfähig bleibt, selbst wenn Teile der Infrastruktur beeinträchtigt sind
{% include padding-id.html id=9 %}

Geografische Verteilung

  • Nutzung der geografischen Verteilung zur Maximierung von Zuverlässigkeit und Verfügbarkeit
  • Datenbanken werden über mehrere geografisch getrennte Regionen hinweg repliziert
  • Minimiert das Risiko eines vollständigen Ausfalls
  • Der Datenzugriff wird nahtlos auf andere Standorte umgeleitet, ohne dass der Endbenutzer eine Unterbrechung bemerkt
